# GLM5.2 EP4 on GB300

> **TL;DR:** DP4/EP4 (`--moe-topo ep4`) brings the EP high-throughput layout to 4×GB300: 64 whole routed experts per rank (~188 GiB weights/rank of 284 GiB), a second DeepEP shim instantiation (`glm52_ep4_deepep_*`), and a new **weight-only routed-expert chain** replacing the sm_90a-only DeepGEMM masked GEMMs — bf16 recv rows × on-the-fly fp8 dequant on `mma.m16n8k16`, running directly on the DeepEP aligned receive layout (no activation re-quant, no masked relayout, no output remap). Bring-up gates green 2026-07-12: 4-bucket whole-step graph capture, greedy byte-determinism ×2, 8-way concurrent greedy (per-shape identical), sampled fallback, **layer-6 EP4 oracle 63/64 × 4 buckets** (after fixing the FlashMLA sm100 UE8M0 KV-scale contract — see lessons doc — which also un-broke attention accuracy for every heads-64 GLM5.2 path on GB300). First numbers (untuned): bs=1 TPOT `23.1 ms`; **c=32 output `559 tok/s`, TPOT p50 `39.05 ms` / p99 `39.18 ms`**. GEMV chain right-sized 2026-07-13 (GB300 split-K table + fused reduce-SwiGLU): b8 step p50 `40.68→40.03 ms`, b1 flat — falsified attempts documented below. MLA q_a|kv_a horizontal fusion (one packed launch + split-reduce): b8 `40.03→39.11 ms` (−2.3%). Opt-in warm-cache pinned staging (`--glm52-weight-staging`) cut full four-rank weight-load wall time from `54.985→17.175 s` and engine/HTTP-ready from `63.192→25.418 s` (two-run means). It remains off by default because a forced-cold network-filesystem A/B regressed HTTP-ready from `144.002→221.887 s`; residency-aware automatic selection is follow-up work. Needs `EP_DISABLE_GIN=1` on NIC-less nodes.
>
> **Last touched:** 2026-07

## Scope

EP4 is the four-GPU counterpart of EP8: a throughput topology (4 logical DP
ranks × 8 slots = 32 concurrent requests), not a bs=1 latency play — TP4
remains the low-latency four-GPU topology (`tp4-gb300-bringup.md`; its "EP4
out of scope" verdict was about an attention-TP4 + expert-EP4 *hybrid* for
bs=1, a different design than this DP4/EP4 topology).

| Topology | Devices | TP / DP / EP | Expert placement | Concurrency |
| --- | ---: | --- | --- | ---: |
| EP8 | 8 | 1 / 8 / 8 | 32 whole experts/rank | 64 |
| EP4 | 4 | 1 / 4 / 4 | 64 whole experts/rank | 32 |
| TP4 | 4 | 4 / 1 / 1 | 1/4 slice of all experts | 8 |

Launch: `--moe-topo ep4` (dp-size defaults to 4, tp-size 1). On this tray the
server needs `EP_DISABLE_GIN=1` (container has no NIC; NCCL GIN probe fails at
`ctx_create` otherwise) plus the usual `OPENINFER_NCCL_ROOT` /
`LD_LIBRARY_PATH` for NCCL ≥ 2.30.4.

## Design

### What is shared with EP8

The DP coordinator protocol, scheduler, KV paging/prefix cache, MLA path
(full 64-head → FlashMLA sparse decode, assembled `sm_100f` on Blackwell),
router, and the per-layer collective calling contract are unchanged — the
model crate's EP surface is one enum (`Glm52MoeEpState`) whose variant is
picked at launch. `global_tokens = ep_ranks × bucket` (32 max for EP4) flows
from a stored model field instead of the old EP8 constant.

### EP4 DeepEP shim instantiation

`csrc/deepep/deepep_shim_glm52_ep4.cu` instantiates the same parameterized
shim body with `kNumRanks=4, kNumLocalExperts=64` under `glm52_ep4_deepep_*`
symbols (the two instantiations co-reside in the binary; `Glm52Ep4DeepEp` is
one more `deepep_abi!` stamp). Device facts stay at the EP8/H200 values:
GB300's `sharedMemPerBlockOptin` equals H200's 232448 and 152 SMs ≥ the baked
132 floor.

### Weight-only routed-expert chain (`glm52_moe_ep_wo.cu`)

The EP8 chain's DeepGEMM masked grouped GEMM is sm_90a-only; GB300 is SM103.
Instead of porting DeepGEMM's SM100 1d1d path (tcgen05 + UE8M0 scale requant
of the whole expert slab), EP4 runs a weight-only chain built from the proven
`glm52_moe_gemv.cu` batched-mma treatment:

```text
dispatch → psum → tile list → W13 mma → silu → W2 mma·route_weight → combine
```

- **Tile list**: DeepEP's aligned receive layout is already expert-major with
  64-aligned segment starts, and 8 | 64 — an 8-row tile never straddles an
  expert. A one-block metadata kernel scans `psum_expert` into a compact
  `(expert, row base, rows ≤ 8)` work list + device tile count (device-trap
  on cross-rank token-count disagreement, same contract as EP8's metadata
  kernel).
- **Masked grouped mma**: per (tile × 128-row output chunk) block, bf16 recv
  rows × on-the-fly fp8→bf16 dequant (exact) on `mma.m16n8k16`, f32
  accumulation with per-128-column scale application in fixed order —
  deterministic per bucket. Full-K single pass, no k-split scratch. Grids are
  shaped at the host-known worst-case tile count (96 at 32 global tokens);
  blocks past the device count retire on one read — CUDA-graph capturable.
- **What disappears**: fp8 activation re-quant, the masked-slab relayout, and
  the masked→aligned output remap (W2 writes the aligned slots
  `decode_combine` addresses directly). Six kernels between collectives
  become four, and the activations skip a quantization step entirely
  (numerically *tighter* than EP8's fp8×fp8, not looser).

Arch note: the chain is arch-portable (plain bf16 mma) — nothing about it is
Blackwell-specific; EP8 keeps its measured DeepGEMM chain on H200.

Known cost vs a true masked M-tile GEMM: an expert with more than 8 rows
re-reads its weights once per 8-row tile (~+15% expert weight traffic at
c=32; none at low concurrency). That is the first optimization lever if
profiles demand it.

### Memory

64 experts × 75 MoE layers × ~37.7 MB ≈ 181 GiB expert slab + non-expert
stack = **188 GiB weights/rank**; the VRAM-derived context cap landed at
`max_model_len=206080` with ~2 GiB post-build headroom per rank.

### Warm-page-cache weight upload

The rank loader now stages checkpoint bytes through two 32 MiB pinned host
buffers per rank and overlaps filling one buffer from the safetensors mmap
with the previous buffer's H2D DMA. Four persistent workers per rank avoid
creating and joining threads for each of GLM5.2's 30,660 uploads/rank.
The workers live in a rank-local Rayon pool; each fill uses scoped borrowed
source/destination slices, so the API cannot return while a worker still
borrows an mmap or pinned slot. Sub-1 MiB tensors, including the 3 KiB expert
scales, use a direct fill to avoid task fan-out overhead. The existing
shard-mmap lifetime guard and final expert-major device placement are
unchanged.

Pinned staging is opt-in with `--glm52-weight-staging`; omitting the flag uses
the original pageable-mmap path. This is also the production A/B switch. EP4
pins 256 MiB total (2 × 32 MiB × 4 ranks), EP8 pins 512 MiB, and EP16 pins
1 GiB.

Configuration:

| Parameter | Value | Scope / reason |
| --- | ---: | --- |
| Pinned slots | 2/rank | One CPU fill can overlap one H2D DMA |
| Slot size | 32 MiB | 64 MiB pinned/rank |
| Fill workers | 4/rank | Persistent for GLM5.2's tensor count |
| Parallel-fill cutoff | 1 MiB | Avoid worker fan-out for 3 KiB expert scales |
| EP4 host total | 256 MiB, 16 workers | 4 ranks × the per-rank resources above |

The results below were collected on 4×GB300 host B:

| Host characteristic | Value |
| --- | --- |
| GPUs | 4 × NVIDIA GB300, 284,208 MiB each |
| CPUs | 2 sockets × 72 Neoverse-V2 cores |
| Host memory | 956 GiB |
| NUMA topology | 34 nodes exposed by the platform |
| Checkpoint storage | Network filesystem |

The final constants come from this small same-host, warm-cache directional
sweep. Every sample is a fresh process that loaded four independent
188.006 GiB rank layouts, created the EP4 contexts, bound HTTP, and answered
`GET /v1/models`. The table reports the critical-rank load wall and
engine-ready wall:

| Slot | Workers/rank | Samples | Weight load | Engine ready |
| ---: | ---: | ---: | --- | --- |
| 64 MiB | 2 | 1 | `29.51 s` | `37.726 s` |
| 64 MiB | 4 | 3 | `18.74 / 15.61 / 19.32 s` (median `18.74`) | `26.978 / 23.857 / 27.566 s` (median `26.978`) |
| 64 MiB | 8 | 1 | `33.85 s` | `42.218 s` |
| 32 MiB | 4 | 2 | `17.37 / 19.82 s` (midpoint `18.60`) | `25.655 / 28.032 s` (midpoint `26.844`) |

Four workers clearly beat 2 and 8 in this directional worker sweep. The
32/64 MiB distributions overlap and their center differs by less than 1%;
32 MiB is retained because it halves pinned footprint while adding only 143
H2D chunks/rank (`30,946` versus `30,803`). This is a host-specific measured
trade-off, not a claim that 32 MiB is universally optimal.

Uploader A/B on 2026-07-23 used EP4 rank 0 on earlier 4×GB300 host A, with
the process bound to GPU0's CPU NUMA node. The checkpoint was on a network
filesystem. Before timing, the benchmark evicted the checkpoint and
read exactly the rank-selected pages with sequential readahead disabled;
`mincore` reported `188.195/188.195 GiB` resident both before and after the
run. Each sample streamed the real rank load plan — 188.006 GiB logical
payload, 30,660 tensors, 30,803 H2D chunks — through a reusable 64 MiB
device scratch allocation:

| Uploader | Three interleaved samples | Median | Median throughput |
| --- | --- | ---: | ---: |
| pageable mmap | `32.043 / 24.558 / 24.765 s` | `24.765 s` | `7.592 GiB/s` |
| pinned double buffer | `11.913 / 11.850 / 11.962 s` | `11.913 s` | `15.782 GiB/s` |

The median uploader speedup is **2.079×** (`−12.852 s`, `−51.9%`). This is a
full-rank source/H2D-stream measurement under a proven 100% warm page-cache
condition, not a full server-startup number: the scratch harness deliberately
does not retain all 188 GiB in VRAM. The resident-loader and full EP4
HTTP-ready gates below close that measurement boundary.

A clean-room repeat on 4×GB300 host B (2026-07-24) first evicted and warmed the
same EP4 rank-0 selection; `mincore` again proved
`188.195/188.195 GiB` resident. Three interleaved pageable/pinned samples gave
`72.734 / 60.987 / 60.790 s` versus `9.699 / 9.528 / 9.671 s`. The pageable
path is host-sensitive and was much slower than on earlier 4×GB300 host A,
while pinned staging was faster and tightly grouped. After replacing the
experimental raw-address worker jobs with the scoped-slice Rayon pool, three
pinned samples were
`9.519 / 9.552 / 9.513 s` (median `9.519 s`, `19.751 GiB/s`), within 1.6% of
the preceding implementation. Against the same-session pageable median this
is **6.407×**, but the cross-host conclusion is deliberately narrower:
pinned staging reproduces and the ownership-safe pool does not regress; the
host B pageable anomaly is not a portable speedup claim.

The final EP4 resident-loader gate retained all `188.006 GiB` in the real
independent-`CudaSlice` model layout. Two interleaved pairs were
`61.304 / 60.939 s` pageable and `9.942 / 10.153 s` pinned, giving midpoint
`61.122 → 10.048 s` (**6.083×**). All 30,660 tensors and their final byte
placements passed the load-plan byte-count check. This closes the gap between
the reusable-scratch uploader measurement and the production rank loader;
it remains a one-rank gate, not a four-rank HTTP-ready measurement.

A subsequent full four-rank EP4 launch on host B closed that boundary. After
one untimed pass warmed every rank's selected pages, two interleaved
pageable/pinned launches each loaded four independent 188.006 GiB resident
layouts, created the EP4 contexts, bound the HTTP server, and answered
`GET /v1/models`. The per-launch critical-rank weight times were
`54.45 / 55.52 s` pageable versus `18.74 / 15.61 s` pinned; engine-ready
times were `62.636 / 63.747 s` versus `26.978 / 23.857 s`. Thus the two-run
means are `54.985 → 17.175 s` for weight load (**3.202×**) and
`63.192 → 25.418 s` for engine/HTTP-ready (**2.486×**, `−37.774 s`).
Each shutdown returned all four GPUs to 0 MiB. This is the end-to-end EP4
startup result; the larger one-rank speedup above is retained as attribution,
not substituted for it.

A forced-cold follow-up used `POSIX_FADV_DONTNEED` before each launch;
`mincore` proved only `0.050/188.195 GiB` and `0.034/188.195 GiB` of rank
0's selected pages resident. Pageable reached weight-load/engine-ready in
`135.76/144.002 s`, while pinned took `213.58/221.887 s` (one matched pair).
Pinned `sync` remained about 0.5 seconds: the regression was in
`mmap_deser_copy`, consistent with four ranks × four fill workers concurrently
faulting network-filesystem pages and disrupting cold sequential throughput.
Therefore pinned staging is deliberately not the default. Automatic
residency-aware selection should be implemented and re-gated separately before
changing that default.

The same host B session also measured the target EP8 rank-0 plan directly,
rather than extrapolating from EP4. `mincore` proved
`103.826/103.826 GiB` resident; the logical payload was `103.610 GiB` across
16,260 tensors and 16,403 H2D chunks. Three interleaved scratch-stream samples
were `28.273 / 28.003 / 28.248 s` pageable versus
`5.254 / 5.200 / 5.254 s` pinned: median `28.248 → 5.254 s` (**5.376×**,
`3.668 → 19.720 GiB/s`). The real resident loader, retaining every independent
`CudaSlice` in its final model layout, gave `28.269 / 28.149 s` pageable and
`5.476 / 5.530 s` pinned in two interleaved pairs. Allocation and final-layout
bookkeeping therefore added only about 0.2 seconds to the pinned path on this
host. That result does not justify a device arena or allocation-first two-pass
refactor as a performance requirement for this change; such a refactor may
still be useful separately for fail-fast validation and observability.

## Bring-up validation (2026-07-12, earlier 4×GB300 host A)

- Kernel smoke (`openinfer-kernels/tests/glm52_moe_ep_wo_smoke.rs`, single
  GPU): tile list exact; W13/W2 mma vs f32 host reference max rel err
  `3.8e-3` (bf16-store floor); alignment-gap rows untouched; SiLU exact.
- Focused unit tests: topology shapes (`ep4` = DP4/EP4/64-local), EP4
  resident load plan carries whole experts 64..128 on rank 1, server config
  accepts `ep4`/rejects bad dp-size.
- Full launch: 4 × 188 GiB weight load (~140 s warm), EP4 DeepEP contexts up,
  **whole-step graphs pre-captured for all 4 buckets**.
- Greedy byte-determinism ×2 (`" Paris. Currency: Euro. ..."`).
- 8-way concurrent greedy: outputs identical *per step shape*; the variants
  across shapes are the accepted cross-bucket FP-association contract
  (`continuous-batching.md`), not an EP4 regression.
- Sampled fallback (temperature 1, top-p 0.95) coherent.
- Layer-6 EP4 oracle gate (`layer_moe_ep4_oracle_gate`, 4 GPUs): replays the
  gemv-precision HF probe constants through the real 4-rank collective —
  **63/64 probes within `2.7e-4` at every global-token bucket** (one outlier
  at 1.4× tol, 4 allowed), identical across buckets (engine bit-stable per
  shape). Getting here required fixing a latent GB300 attention bug that had
  nothing to do with EP4: the FlashMLA sm100 kernel truncates fp8 KV-cache
  scales to powers of two, so the cache writer must emit UE8M0 scales — see
  `docs/lessons/flashmla-sm100-ue8m0-kv-scales.md`. With that fix the MLA
  oracle gate and the dense layer-0 gate are 64/64 on GB300 as well (both
  were silently failing before), and the weights-free
  `flashmla_sparse_vs_reference_gate` pins the kernel against an f64
  reference on any arch.

## First performance numbers (untuned)

`vllm bench serve`, random 64-in, 128-out, ignore-eos, same-day single
session:

| Shape | TTFT p50 | TPOT p50 | TPOT p99 | Output tok/s |
| --- | ---: | ---: | ---: | ---: |
| c=1, n=8 | `221 ms` | `23.14 ms` | — | `40.5` |
| c=32, n=128 | `2361 ms` | `39.05 ms` | `39.18 ms` | **`558.7`** |

Re-measured 2026-07-12 after the UE8M0 KV-scale fix and the route-weight
move: c=1 TPOT p50 `23.2 ms`, c=32 TPOT p50 `39.1 ms` / p99 `39.3 ms`,
`554.8 tok/s` — identical within run noise, as expected (same kernels; the
fix only changes the scale rounding inside the cache-write quant).

Context: TP4 on the same box serves bs=1 at ~`9.4-9.7 ms` TPOT but holds at
most 8 requests; EP4 trades bs=1 latency for 32-slot throughput. No
optimization pass has run — no nsys attribution, no MoE grid tuning, no
launch-ahead audit. Do not quote these as the topology's ceiling, and
re-baseline both engines in one session before any comparison (this tray
drifts across days — see the tp4 doc's 07-11 re-validation).

## Step attribution (2026-07-12, first pass)

nsys cannot trace the whole-step graph replay on this stack (node- AND
graph-level CUDA graph tracing segfault or hang the process — nsys 2026.3 +
CUDA 13 + sm103; profile the EAGER oracle-gate walk instead, which runs the
same per-layer kernel composition without graphs). Two data sources agree:

**Bucket sweep** (`glm52_step_bench --buckets 1,2,4,8`, steps p50):
`23.5 / 28.2 / 35.3 / 42.2 ms`. (An earlier sweep read `28.2 / 35.4 / 41.7 /
41.7` and concluded "bucket 4→8 is free" — that run submitted 2× the intended
streams because `bench_concurrency` used the EP8 rank constant on EP4, so
every label was one bucket off; PR review caught it.) The corrected reading:
a ~23.5 ms fixed floor (matches serving bs=1 TPOT) plus ~5-7 ms per bucket
doubling — strongly sublinear (bucket 4→8 doubles tokens for +19% step
time), so throughput scales well with concurrency but not for free.

**Eager per-layer GPU busy** (rank 0, fixed bucket-1 shapes, nsys on
`layer_moe_ep4_oracle_gate`): ~330 µs/layer = WO expert chain 107 µs (mma
47.6 µs/GEMM — ~70% of HBM peak on the hit-expert weight sweep) + attention
projections/shared-expert GEMVs 87 µs (~2.5 TB/s effective — rows<=8
weight-only GEMV is the least BW-efficient stage) + DeepEP dispatch/combine
62 µs + MLA/indexer 49 µs + misc 23 µs. x75 layers ~= 24.8 ms, matching the
sweep's ~23.5 ms bucket-1 floor (eager launches inflate kernel durations
slightly vs graph replay).

Levers, in order: (1) raise `GLM52_MAX_BATCH_PER_RANK` past 8 — each bucket
doubling costs ~+6 ms (~+16-19%) for 2x tokens, so deeper slots still buy
strongly sublinear-cost throughput (validate the trend holds at 16/32);
(2) attention/shared GEMV bandwidth at small rows (26% of the step, 31% BW
efficiency); (3) a2a is only 19% — overlap work exists but is not the
headline; (4) remember DP4 replicates all attention/shared weights per rank
(every rank re-reads ~700 MB/layer-equivalent each step) — that is the
structural reason EP wants large global batch.

## b8 GEMV-chain right-sizing (2026-07-13, branch `feat/glm52-b8-gemv-chain`)

The bucket-8 step spends ~7.6 ms/step in the batched weight-only GEMV family
(vs ~1.8 ms roofline). Two shipped changes, gated on this box (locked clocks,
`glm52_step_bench --buckets 8` / `--buckets 1`, 60 steps):

| config | b8 p50 / p99 | b1 p50 / p99 |
| --- | --- | --- |
| main (a2d0a8c) | 40.68 / 41.6 | 22.62 / 22.7 |
| + GB300 split-K table + fused reduce-SwiGLU | **40.03 / 41.31** | 22.57 / 22.68 |

- **GB300 batch-8 split-K table** (`mma_config`, gated on
  `arch_is_blackwell()`): every whitelisted shape re-swept on GB300 sm_103
  over the full (KSPLIT × NTILES) grid with a standalone harness (one
  `.cu` compiling the production kernel at every template instantiation,
  locked clocks, 512-iteration medians) — pair-time winners like q_a 48/2
  (8.1→7.1 µs) and kv_a 16/1 (6.7→5.5 µs). Hopper tables untouched. The
  H200 picks do NOT transfer (the wall is bytes-in-flight per SM, an arch
  property). Attribution: this table is the bulk of the −0.65 ms; the
  SwiGLU fusion below is ~−0.15 ms of it.
- **Fused reduce-SwiGLU** (`glm52_gemv_reduce_silu_mul`): the packed gate|up
  mma path stops at its f32 k-slice partials and the SwiGLU absorbs the
  fixed-order reduce — bit-identical (sums round to bf16 before the SiLU),
  one launch instead of two, and the `[rows, 2*inter]` bf16 round-trip
  disappears. Wins because silu's grid (rows × inter/256 ≈ 64 blocks) can
  host the reduce work.

**Falsified on this box (don't repeat):**

- *reduce→RMSNorm fusion* (q_a norm + post-attn add+norm absorbing the
  reduce; bit-identical impl, oracle green): b8 40.03→40.84/40.72 — norm
  kernels are block-per-row (grid = 8 blocks on ~148 SMs), so the
  ksplit=16-48 slice reduction collapses onto 8 latency-bound blocks. A
  fusion only wins when the consumer's grid can host the producer's
  parallelism.
- *Five structural GEMV alternatives*, all measured slower or unavailable:
  smem-act + deep prefetch (occupancy backfire), `__launch_bounds__(128,8)`
  (register spills), cp.async staging (`__syncthreads` tax), Triton split-K
  (o_proj 36.8 vs 22.8 µs — the H200 1.3× win does not transfer), DeepGEMM
  fp8 masked (does not exist on SM100; its "fp8" masked entry is an alias
  to fp8×fp4). A pure-read control at the shipping grid geometry reaches
  6.4-6.8 TB/s, so the kernel is at its W8A16 structural optimum — the
  remaining gap is the cvt+mma issue chain under register-squeezed
  occupancy.

Deeper levers (W4A8 weights, DeepEP protocol tax ~4.4 ms/step, masked-mma
small-bucket right-sizing) are roadmap items — issue #668 carries the full
profile data.

## MLA q_a|kv_a horizontal fusion (2026-07-13, branch `feat/glm52-mla-front-fused-gemv`)

After the split-K table, the remaining GEMV account is dominated by
**fixed per-launch tax on small weights**, not bytes: kv_a `[576,6144]`
costs 5.5 µs on ~0.5 µs of bytes. The MLA front's q_a `[2048,6144]` and
kv_a share the same hidden-state input, so packing the two weights row-wise
into a `[2624,6144]` twin turns two launches into one — locked-clock sweep
says **7.3 µs fused (16/1) vs 7.1 + 5.5 separate**: the kv_a bytes ride
free inside the q_a launch's ramp.

| config | b8 p50 / p90 / p99 | b1 p50 / p99 |
| --- | --- | --- |
| main (16c2de8, post-#671) | 40.03 / — / 41.31 | 22.58 / 22.68 |
| + q_a\|kv_a fused GEMV | **39.11 / 39.60 / 39.92** (−2.3%) | 22.68 / 22.81 |

Mechanics:

- The packed twin stops at f32 split-K partials; a new
  `glm52_gemv_split_reduce` kernel de-interleaves them into the bf16
  q_a `[t,2048]` and kv_a `[t,576]` outputs in one pass (fixed ascending-s
  order, same k-slicing as the separate kv_a 16/1 launch → **kv_a output is
  bit-exact vs unfused**; q_a moves 48→16 k-slices, tolerance-level).
- The twin (~15.4 MiB/layer, ~1.17 GiB/rank) is built **only where the
  Blackwell batch-8 mma table routes the packed shape**
  (`glm52_gemv_mma_routes`); Hopper, t==1, and batch-2/4 keep the existing
  separate/pair launches bit-identically. The twin bytes are charged to the
  context-cap budget up front — they allocate after the free-VRAM probe, and
  an unledgered 1.17 GiB eats the post-build 1 GiB headroom gate (this fired
  for real on the first gate run).
- b1 lands 22.68 vs 22.58 (+0.1 ms): t==1 never touches the fused path; the
  shift is allocator/layout displacement from 1.17 GiB of extra resident
  weights, stable across reruns.
- Coverage: `qa_kva_pack_batch8_parity` (GPU test) checks the packed
  twin + split-reduce against separate launches at t=8; dense/EP4/bookend
  oracles green.

## Pitfalls

- `EP_DISABLE_GIN=1` is required on NIC-less nodes or `ctx_create` fails
  with "NCCL GIN unavailable".
- The EP4 shim asserts `num_ranks == 4` at `ctx_create`; the EP8 and EP4
  shims are distinct symbol sets — passing the wrong unique-id generator
  (`glm52_deepep_unique_id` vs `glm52_ep4_deepep_unique_id`) desyncs NCCL
  init.
- The weight-only GEMM consumes the checkpoint's f32 `[n/128, k/128]`
  weight scales as loaded (no requant). The **KV cache** is the opposite:
  its per-token group scales MUST be UE8M0 (powers of two) or the FlashMLA
  sm100 kernel silently truncates them — the lessons doc above has the full
  story.
- `GLM52_EP_RANKS`/`GLM52_LOCAL_EXPERTS` (8/32) remain in `weights.rs` as the
  manifest-coverage partition and EP8 shim contract only; serving code derives
  counts from `Glm52MoeTopo`.
